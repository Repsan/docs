## What is OpenAI Agents SDK?

The [OpenAI Agents SDK](https://github.com/openai/openai-agents-python) is a lightweight framework for building multi-agent workflows with tools, handoffs, and guardrails. It provides a built-in tracing system that captures agent runs, LLM generations, tool calls, and agent-to-agent handoffs.

## Attributes

The exporter automatically captures:

| Attribute | Source | Description |
|-----------|--------|-------------|
| `model` | `Generation` / `Response` | Model name used for the LLM call |
| `gen_ai.usage.prompt_tokens` | `Generation` / `Response` | Input (prompt) token count |
| `gen_ai.usage.completion_tokens` | `Generation` / `Response` | Output (completion) token count |
| `gen_ai.usage.total_tokens` | `Generation` / `Response` | Total token count |
| `agent_name` | `Agent` span | Name of the agent (e.g. from `Agent(name="...")`) |
| `tool_name` | `Function` span | Name of the tool/function invoked |
| `tool_input` / `tool_output` | `Function` span | Arguments passed and result returned |
| `from_agent` / `to_agent` | `Handoff` span | Source and target agent names for handoffs |
| `triggered` | `Guardrail` span | Whether the guardrail tripped (`true` / `false`) |
| `error_message` | `Function` / `Agent` span | Error message when a span ends in error |

## Traced events

The exporter hooks into the SDK's tracing system and produces the following span types:

| Event | Span type | Description |
|-------|-----------|-------------|
| `trace(...)` / root run | `trace` | Root span for the workflow |
| `Runner.run` | `agent` | One agent run; contains `agent` name and nested spans |
| `LLM API call` | `response` | LLM response; carries `model` and usage |
| `LLM API call` | `generation` | Per-call generation data (`model`, tokens) |
| `Tool invocation` | `function` | Tool/function call with name, `input`, `output`, and timing |
| `Agent handoff` | `handoff` | Agent-to-agent handoff with `from_agent` and `to_agent` |
| `Input/output guardrail` | `guardrail` | Guardrail check; may include `triggered` |

### Handoffs

Agent-to-agent handoffs are traced with full context.

```python
from agents import Agent, Runner, set_trace_processors

billing_agent = Agent(
    name="Billing Agent",

    name="Support Agent",
    instructions="Route billing questions to the billing agent.",
    handoffs=[billing_agent],
)

result = Runner.run_sync(support_agent, "I have a billing question")

```

### Agents as tools

Instead of handoffs (where the new agent takes over), you can use agents as tools ŌĆö the tool agent runs independently and returns its result to the caller.

```python
from agents import Agent, Runner, set_trace_processors, trace
from respan_exporter_openai_agents import RespanTraceProcessor

set_trace_processors([RespanTraceProcessor()])

spanish_agent = Agent(
    name="spanish_agent",
    instructions="You translate the user's message to Spanish",
)

french_agent = Agent(
    name="french_agent",
    instructions="You translate the user's message to French",
)

orchestrator = Agent(
    name="orchestrator",
    instructions="You are a translation agent. Use the provided tools to translate.",
    tools=[
        spanish_agent.as_tool(
            tool_name="translate_to_spanish",
            tool_description="Translate to Spanish",
        ),
        french_agent.as_tool(
            tool_name="translate_to_french",
            tool_description="Translate to French",
        ),
    ],
)

with trace("Translation orchestrator"):
    result = Runner.run_sync(orchestrator, "Translate 'hello' to Spanish and French")
    print(result.final_output)
```

### Guardrails

Run input guardrails in parallel with the agent to quickly reject invalid inputs.

```python
from agents import (
    Agent, Runner, set_trace_processors,
    input_guardrail, GuardrailFunctionOutput,
    InputGuardrailTripwireTriggered, RunContextWrapper,
)
from pydantic import BaseModel
from respan_exporter_openai_agents import RespanTraceProcessor

set_trace_processors([RespanTraceProcessor()])

class MathHomeworkOutput(BaseModel):
    is_math_homework: bool
    reasoning: str

guardrail_agent = Agent(
    name="Guardrail check",
    instructions="Check if the user is asking you to do their math homework.",
    output_type=MathHomeworkOutput,
)

@input_guardrail
async def math_guardrail(context, agent, input):
    result = await Runner.run(guardrail_agent, input, context=context.context)
    output = result.final_output_as(MathHomeworkOutput)
    return GuardrailFunctionOutput(
        output_info=output,
        tripwire_triggered=output.is_math_homework,
    )

agent = Agent(
    name="Support Agent",
    instructions="You are a customer support agent.",
    input_guardrails=[math_guardrail],
)

try:
    result = await Runner.run(agent, "Solve for x: 2x + 5 = 11")
except InputGuardrailTripwireTriggered:
    print("Guardrail triggered ŌĆö input rejected")
```

### Streaming

Stream agent responses with real-time text deltas.

```python
from openai.types.responses import ResponseTextDeltaEvent
from agents import Agent, Runner

agent = Agent(name="Joker", instructions="You tell jokes.")

result = Runner.run_streamed(agent, input="Tell me 3 jokes.")
async for event in result.stream_events():
    if event.type == "raw_response_event" and isinstance(
        event.data, ResponseTextDeltaEvent
    ):
        print(event.data.delta, end="", flush=True)
```

### Parallelization

Run multiple agents concurrently and pick the best result.

```python
from agents import Agent, Runner, ItemHelpers, trace
import asyncio

spanish_agent = Agent(name="translator", instructions="Translate to Spanish")
picker = Agent(name="picker", instructions="Pick the best translation")

with trace("Parallel translation"):
    res_1, res_2, res_3 = await asyncio.gather(
        Runner.run(spanish_agent, "Good morning"),
        Runner.run(spanish_agent, "Good morning"),
        Runner.run(spanish_agent, "Good morning"),
    )

    outputs = [ItemHelpers.text_message_outputs(r.new_items) for r in [res_1, res_2, res_3]]
    best = await Runner.run(picker, f"Translations:\n{chr(10).join(outputs)}")
    print(best.final_output)
```

### Deterministic flows

Sequential agent pipeline with validation gates.

```python
from pydantic import BaseModel
from agents import Agent, Runner, trace

outline_agent = Agent(name="outliner", instructions="Generate a story outline.")

class CheckResult(BaseModel):
    good_quality: bool
    is_scifi: bool

checker_agent = Agent(name="checker", instructions="Judge outline quality.", output_type=CheckResult)
writer_agent = Agent(name="writer", instructions="Write a short story from the outline.")

with trace("Story pipeline"):
    outline = await Runner.run(outline_agent, "A robot discovers emotions")
    check = await Runner.run(checker_agent, outline.final_output)

    if check.final_output.good_quality and check.final_output.is_scifi:
        story = await Runner.run(writer_agent, outline.final_output)
        print(story.final_output)
```

### Dynamic system prompts

Use a function for `instructions` to customize prompts based on runtime context.

```python
from agents import Agent, Runner, RunContextWrapper, set_trace_processors
from respan_exporter_openai_agents import RespanTraceProcessor

set_trace_processors([RespanTraceProcessor()])

class CustomContext:
    def __init__(self, style: str):
        self.style = style

def custom_instructions(
    run_context: RunContextWrapper[CustomContext], agent: Agent
) -> str:
    if run_context.context.style == "haiku":
        return "Only respond in haikus."
    elif run_context.context.style == "pirate":
        return "Respond as a pirate."
    return "Respond normally."

agent = Agent(name="Chat agent", instructions=custom_instructions)

result = await Runner.run(
    agent, "Tell me a joke.", context=CustomContext(style="pirate")
)
print(result.final_output)
```

## How it works

The example uses two Respan integration points with a single API key:

```python
from openai import AsyncOpenAI
from agents import set_default_openai_client
from agents.tracing import set_trace_processors
from respan_exporter_openai_agents import RespanTraceProcessor

# Route all LLM calls through the Respan gateway
client = AsyncOpenAI(api_key=RESPAN_API_KEY, base_url=RESPAN_BASE_URL)
set_default_openai_client(client)

# Export all spans to Respan
set_trace_processors([
    RespanTraceProcessor(
        api_key=RESPAN_API_KEY,
        default_model=RESPAN_MODEL,
    ),
])
```

| Component | Purpose |
|-----------|---------|
| `AsyncOpenAI` client | Routes LLM calls through the Respan gateway ŌĆö no OpenAI key needed. |
| `RespanTraceProcessor` | Hooks into the SDK's tracing system and exports spans to Respan. |
| `default_model` | Fallback model name for spans that don't carry their own (agent, tool, handoff). |

All 12 scenarios run inside a single `with trace("Edge Case Stress Test"):` block, producing one unified trace tree.

## Scenarios

The example defines 8 tools, 2 guardrails, and 8 agents, then composes them into 12 scenarios. Each scenario targets a specific span type or edge case.

### 1. Three-level handoff chain

**Span types:** Handoff, Agent, Response, Generation

A weather question enters the **Triage Agent**, which hands off to **Weather Router**, which hands off to **Weather Detail Agent**. This creates a three-level handoff chain that tests deeply nested `parent_id` propagation and that each handoff span correctly captures `from_agent` and `to_agent`.

```
Triage Agent ŌåÆ (handoff) ŌåÆ Weather Router ŌåÆ (handoff) ŌåÆ Weather Detail Agent ŌåÆ get_weather
```

### 3. Tool error recovery + slow tool timing

**Span types:** Function (error), Function (slow), Function (normal)

The **Resilience Agent** runs three tools in sequence:

1. `slow_database_query` ŌĆö a 3-second async delay that tests span timing accuracy
2. `get_secret_data` ŌĆö always raises `PermissionError`, testing that errored tool spans are captured with `error_bit=1` and `error_message` without crashing the trace
3. `get_weather` ŌĆö a normal tool call to prove the agent recovers after the error

This verifies that one errored span doesn't corrupt sibling spans.

### 4. Structured output with output guardrail

**Span types:** Guardrail, Agent, Response

The **Analysis Agent** returns a structured `QualityOutput` (with `reasoning`, `response`, and `confidence` fields) and has an output guardrail that trips if `confidence < 0.2`. This tests structured output serialization and that the guardrail span correctly links to the agent span via `parent_id`, even though it arrives after the agent span completes.

**Span types:** Function wrapping Agent + Response + Generation

The **Orchestrator** uses two agents-as-tools via `.as_tool()`:

- `translate_to_french` ŌĆö wraps the **Translator** agent
- `summarize` ŌĆö wraps the **Summarizer** agent

Each Function span wraps an entire sub-agent run with its own Agent, Response, and Generation spans inside. This creates recursive span nesting that tests the exporter's ability to handle deeply nested trees.

### 6. Large payload (~50KB tool output)

**Span types:** Function, Agent, Response

The **Report Agent** calls `generate_large_report`, which returns approximately 50KB of text. This tests payload size limits on the exporter's HTTP POST and that the batch processor handles oversized items gracefully.

### 7. Unicode / emoji / special character encoding

**Span types:** Function (x4), Agent, Response

A dedicated **Unicode Agent** calls `get_localized_greeting` for four languages ŌĆö Japanese (with emoji and CJK), Arabic (RTL text), emoji (family emoji, flag sequences, regional indicators), and special characters (quotes, slashes, angle brackets, tabs). ### Tools reference

| Tool | Edge case | What it tests |

| `get_weather` | Normal return | Baseline string output |

| `slow_database_query` | 3-second async delay | Span timing accuracy, batch processor timeout |
| `get_secret_data` | Always raises `PermissionError` | Error span capture without crashing the trace |
| `generate_large_report` | ~50KB output | Payload size limits |
